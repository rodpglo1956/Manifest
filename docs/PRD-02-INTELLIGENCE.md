# PRD-02: Intelligence Layer

**Manifest (by Glo Matrix)** | Phase 2 | Weeks 9-16 | v1.1 | March 2026

---

## 1. What this phase covers

Phase 2 adds the intelligence layer that separates Manifest from a basic dispatch tool. Marie AI becomes the operations assistant. Smart routing suggests optimal driver-load assignments. Predictive alerts warn about issues before they become problems. The analytics foundation gives operators real visibility into their business.

**Depends on:** PRD-01 (Foundation) must be complete. Auth, loads, dispatch, drivers, invoicing, and the dashboard skeleton must be operational.

## 2. Marie AI operations assistant

### 2.1 What Marie does

Marie is an AI assistant embedded in Manifest that understands the operator's entire operation. She reads from the same Supabase database every other feature uses. She does not have her own data store. She queries the production tables and returns answers in natural language.

**Proactive alerts (Marie pushes to you):**
- "Load #4521 pickup is in 2 hours and the assigned driver is 180 miles away. You may need to reassign."
- "3 invoices are now overdue. Total outstanding: $8,400."
- "Driver Martinez has not updated his status in 6 hours. Last known status: in transit."

**On-demand queries (you ask Marie):**
- "How many loads did we deliver this week?"
- "What is our average revenue per mile this month?"
- "Which driver has the most loads this quarter?"
- "Show me all loads going to Atlanta this week."

**Action execution (Marie does things):**
- "Create a load from Dallas to Houston, dry van, $2,800 flat rate, picking up tomorrow at 8 AM."
- "Dispatch driver Johnson to load #4530."
- "Generate an invoice for load #4521."

### 2.2 Architecture

Marie runs on Railway as a standalone service. She receives queries from the Manifest frontend via API and responds in real-time.

```
User (Manifest UI) -> /api/marie/query -> Railway (Marie service) -> Claude API -> Supabase (read data) -> response
```

**Key constraints:**
- Marie reads from Supabase using the same RLS policies. She never sees data outside the requesting user's org.
- Marie's Claude API calls include the org context and user role. She will not execute admin actions for a driver-role user.
- Marie does not store conversation history in a separate table. Each query is stateless. Context is rebuilt from the database on every request.
- Marie's responses are logged in a `marie_queries` table for analytics and debugging.

### 2.3 Schema

```sql
create table marie_queries (
  id uuid primary key default gen_random_uuid(),
  org_id uuid not null references organizations(id),
  user_id uuid not null references auth.users(id),
  query_text text not null,
  response_text text not null,
  query_type text, -- 'question', 'action', 'alert_response'
  tokens_used integer,
  latency_ms integer,
  model text, -- which Claude model was used
  success boolean default true,
  error_message text,
  created_at timestamptz default now()
);

-- Proactive alerts generated by edge functions, delivered via Marie
create table proactive_alerts (
  id uuid primary key default gen_random_uuid(),
  org_id uuid not null references organizations(id),
  alert_type text not null, -- 'late_pickup', 'overdue_invoice', 'driver_silent', 'dispatch_conflict', 'eta_risk'
  severity text not null, -- 'info', 'warning', 'critical'
  title text not null,
  message text not null,
  related_entity_type text, -- 'load', 'driver', 'invoice', 'dispatch'
  related_entity_id uuid,
  acknowledged boolean default false,
  acknowledged_by uuid references auth.users(id),
  created_at timestamptz default now()
);

alter table marie_queries enable row level security;
alter table proactive_alerts enable row level security;

create policy "org_marie_queries" on marie_queries
  for all using (org_id = (select org_id from profiles where id = auth.uid()));
create policy "org_proactive_alerts" on proactive_alerts
  for all using (org_id = (select org_id from profiles where id = auth.uid()));
```

### 2.4 Marie UI

**Command mode: Marie chat panel**
- Slide-out panel on the right side of any page (triggered by floating Marie button)
- Text input at bottom, conversation thread above
- Marie responses include inline action buttons when relevant ("Dispatch this driver", "View this load", "Generate invoice")
- Alert badge on Marie button shows unacknowledged proactive alert count

**Driver PWA: Marie simplified**
- Small chat input on dashboard

**Owner-Operator mode: Full Marie access**

## 3. Smart routing

### 3.1 What it does

When a dispatcher creates a dispatch (assigns a driver to a load), smart routing provides a ranked recommendation of which drivers are the best fit based on proximity, availability, equipment match, and historical performance.

### 3.2 Ranking factors

| Factor | Weight | Data source |
|---|---|---|
| Proximity to pickup | 30% | Driver's last known location (from last load delivery or manual check-in) vs pickup address |
| Availability | 25% | Driver status (active + not on a current load), HOS remaining (Phase 3) |
| Equipment match | 20% | Driver's assigned vehicle type vs load's required equipment type |
| On-time performance | 15% | Historical on-time delivery percentage from load_status_history |
| Lane familiarity | 10% | How many times the driver has run this origin-destination pair before |

### 3.3 Implementation

Smart routing is an API endpoint, not a separate service. It runs server-side in a Next.js API route.

```
POST /api/dispatch/suggest
Body: { load_id: "..." }
Response: { suggestions: [{ driver_id, score, factors: { proximity, availability, equipment, performance, lane } }, ...] }
```

The dispatch UI shows suggestions as a ranked list. The dispatcher can accept a suggestion or override with manual selection.

### 3.4 UI

**Command mode: /dispatch (enhanced)**
- When assigning a driver, a "Suggested" tab appears showing ranked driver recommendations
- Each suggestion shows the driver name, score, and a breakdown of why they ranked
- One-click assign from the suggestion list
- "Override" button to manually select a different driver

## 4. Predictive alerts

### 4.1 What they do

Edge functions run on schedules and analyze operational data to detect problems before they impact the business. Each alert generates a proactive_alerts record and optionally triggers a push notification.

### 4.2 Alert types

| Alert | Trigger | Schedule | Severity |
|---|---|---|---|
| Late pickup risk | Driver is > 100 miles from pickup with < 3 hours until pickup window | pg_cron: every 30 min | warning/critical based on time |
| Driver gone silent | Driver has not updated load status in > 4 hours while in_transit | pg_cron: every hour | warning at 4h, critical at 8h |
| Overdue invoice | Invoice due_date < today, status still 'sent' | pg_cron: daily 8 AM | warning at 1-7 days, critical at 7+ |
| Dispatch conflict | Two loads assigned to the same driver with overlapping pickup windows | On dispatch creation (API middleware) | critical |
| ETA risk | Estimated delivery time > delivery window end time (based on distance + average speed) | pg_cron: every hour for in_transit loads | warning |
| Unassigned load | Load is 'booked' with pickup_date within 24 hours and no dispatch | pg_cron: every 2 hours | warning at 24h, critical at 12h |

### 4.3 Edge functions

| Function | Schedule | Purpose |
|---|---|---|
| `alert-late-pickup` | pg_cron: every 30 min | Checks proximity of assigned drivers to upcoming pickups |
| `alert-driver-silent` | pg_cron: every hour | Checks last status update time for in-transit loads |
| `alert-eta-risk` | pg_cron: every hour | Calculates ETAs for in-transit loads, flags delivery window risks |
| `alert-unassigned-loads` | pg_cron: every 2 hours | Finds booked loads with pickup < 24h and no dispatch |
| `alert-overdue-invoices` | pg_cron: daily 8 AM | Already exists from Phase 1, enhanced to write proactive_alerts |

### 4.4 Notification delivery

Proactive alerts are delivered three ways:
1. **In-app:** Badge count on Marie button, alert feed in dashboard
2. **Push notification:** Web Push API for Driver PWA and desktop browsers
3. **Marie digest:** Marie summarizes unacknowledged alerts when user opens the chat panel

## 5. Analytics foundation

### 5.1 What it does

Phase 2 builds the data aggregation layer. The full reporting suite (PDF export, CSV, trend charts) comes in Phase 4. Phase 2 delivers the dashboard charts and the data infrastructure behind them.

### 5.2 Schema

```sql
-- Daily operational snapshots (generated by edge function)
create table daily_snapshots (
  id uuid primary key default gen_random_uuid(),
  org_id uuid not null references organizations(id),
  snapshot_date date not null,
  loads_booked integer default 0,
  loads_delivered integer default 0,
  loads_canceled integer default 0,
  revenue numeric(12,2) default 0,
  total_miles integer default 0,
  revenue_per_mile numeric(8,2) default 0,
  on_time_deliveries integer default 0,
  total_deliveries integer default 0,
  on_time_percentage numeric(5,2) default 0,
  active_drivers integer default 0,
  invoices_generated integer default 0,
  invoices_paid numeric(12,2) default 0,
  created_at timestamptz default now(),
  unique(org_id, snapshot_date)
);

alter table daily_snapshots enable row level security;
create policy "org_snapshots" on daily_snapshots
  for select using (org_id = (select org_id from profiles where id = auth.uid()));
```

### 5.3 Edge function

**daily-snapshot-generator (pg_cron: daily at 1 AM)**
- Aggregates previous day's data from loads, dispatches, invoices
- Calculates revenue, miles, on-time percentage, driver utilization
- Writes one row per org to daily_snapshots
- Backfill script available for historical data

### 5.4 Dashboard charts (Command mode)

Phase 2 adds these charts to the existing dashboard:

- **Revenue trend:** Line chart, last 30 days (from daily_snapshots)
- **Load volume:** Bar chart, loads booked vs delivered per week
- **On-time performance:** Gauge chart showing current month percentage
- **Revenue per mile trend:** Line chart, last 30 days

Charts built with Recharts (already in the stack from DESIGN-SYSTEM.md).

## 6. Enhanced dispatch board

### 6.1 What changes

Phase 1 dispatch is a basic list with manual assignment. Phase 2 upgrades it:

- **Map view:** Show unassigned loads as pins on a map, available drivers as different-colored pins. Visual proximity matching.
- **Timeline view:** Gantt-style view of driver schedules. Shows current load, next load, gaps.
- **Smart routing integration:** Suggestion panel appears when selecting a load to dispatch.
- **Conflict detection:** Real-time warning if dispatcher tries to assign a driver who already has an overlapping load.

### 6.2 API additions

| Route | Method | Purpose |
|---|---|---|
| `/api/marie/query` | POST | Send query to Marie, receive response |
| `/api/marie/alerts` | GET | List proactive alerts |
| `/api/marie/alerts/[id]/acknowledge` | POST | Acknowledge alert |
| `/api/dispatch/suggest` | POST | Get smart routing suggestions for a load |
| `/api/dispatch/conflicts` | POST | Check for scheduling conflicts before assignment |
| `/api/analytics/snapshots` | GET | Daily snapshot data for charts |
| `/api/analytics/summary` | GET | Aggregated summary stats for dashboard |

## 7. Push notifications

### 7.1 Implementation

Web Push API with a service worker. Notification permission requested after first successful login (not on signup page).

### 7.2 Notification types

| Type | Recipient | Trigger |
|---|---|---|
| New dispatch assigned | Driver | Dispatch created for their driver_id |
| Load status changed | Dispatcher who created the dispatch | Driver updates status |
| Proactive alert | Admins + dispatchers | Edge function generates critical alert |
| Invoice paid | Admin | Invoice status changes to 'paid' |
| Driver accepted/rejected dispatch | Dispatcher | Driver responds to assignment |

### 7.3 Preferences

Each user can toggle notification types on/off in `/settings`. Stored in a `notification_preferences` jsonb column on profiles.
