# Phase 5: Marie AI & Smart Routing - Research

**Researched:** 2026-03-25
**Domain:** AI assistant integration (Claude API), smart routing algorithm, chat UI
**Confidence:** HIGH

## Summary

Phase 5 introduces Marie, an AI operations assistant backed by the Claude API, and a smart routing algorithm for driver-load matching. Marie is implemented as a Next.js API route (`/api/marie/query`) that proxies to the Claude Messages API with tool_use for executing actions (create load, dispatch driver, generate invoice). Smart routing is a standalone API endpoint (`/api/dispatch/suggest`) that scores drivers against a load using 5 weighted factors.

The core technology is `@anthropic-ai/sdk` v0.80.0 for Claude API integration. Marie's architecture is stateless -- each request rebuilds context from the database, sends it as the system prompt with org-scoped data, and defines tools that map to existing server actions. Streaming is used for the chat UX via `client.messages.stream()` with SSE forwarding through a Next.js API route. Smart routing uses city/state string matching for proximity approximation (no GPS coordinates needed).

**Primary recommendation:** Use `@anthropic-ai/sdk` directly with `client.messages.stream()` for Marie, implementing a manual tool-use loop in the API route. Do NOT use the Vercel AI SDK -- the project has zero dependency on it and adding it would be unnecessary complexity. For streaming to the client, use a `TransformStream` in the Next.js API route that forwards text deltas as SSE events.

### Pattern 1: Claude API Tool Use Loop (Non-Streaming for Actions)
**What:** When Marie needs to execute actions, use a non-streaming tool_use loop. Stream only the final text response.
**When to use:** All Marie queries that may involve tool calls (create load, dispatch, invoice).
**Example:**
```typescript
// Source: Anthropic official docs - tool use implementation
import Anthropic from '@anthropic-ai/sdk'

const client = new Anthropic()

// Define tools with JSON Schema input_schema
const tools: Anthropic.Tool[] = [
  {
    name: 'create_load',
    description: 'Create a new load in the system',
    input_schema: {
      type: 'object' as const,
      properties: {
        pickup_city: { type: 'string', description: 'Pickup city' },
        pickup_state: { type: 'string', description: 'Pickup state abbreviation' },
        delivery_city: { type: 'string', description: 'Delivery city' },
        delivery_state: { type: 'string', description: 'Delivery state abbreviation' },
        equipment_type: { type: 'string', enum: ['dry_van', 'reefer', 'flatbed', 'sprinter', 'box_truck'] },
        rate_amount: { type: 'number', description: 'Rate in dollars' },
        rate_type: { type: 'string', enum: ['flat', 'per_mile'], description: 'Rate type' },
      },
      required: ['pickup_city', 'pickup_state', 'delivery_city', 'delivery_state', 'rate_amount'],
    },
  },
  // ... dispatch_driver, generate_invoice tools
]

// Tool use loop pattern
async function processMarieQuery(messages: Anthropic.MessageParam[], systemPrompt: string) {
  let response = await client.messages.create({
    model: 'claude-sonnet-4-20250514',
    max_tokens: 1024,
    system: systemPrompt,
    tools,
    messages,
  })

  // Loop while Claude wants to use tools
  while (response.stop_reason === 'tool_use') {
    const toolUseBlocks = response.content.filter(b => b.type === 'tool_use')
    const toolResults: Anthropic.ToolResultBlockParam[] = []

    for (const block of toolUseBlocks) {
      const result = await executeTool(block.name, block.input)
      toolResults.push({
        type: 'tool_result',
        tool_use_id: block.id,
        content: JSON.stringify(result),
      })
    }

    // Send tool results back
    messages.push({ role: 'assistant', content: response.content })
    messages.push({ role: 'user', content: toolResults })

    response = await client.messages.create({
      model: 'claude-sonnet-4-20250514',
      max_tokens: 1024,
      system: systemPrompt,
      tools,
      messages,
    })
  }

  return response
}
```

### Pattern 2: Streaming Text Response via SSE
**What:** Stream Marie's text response to the client using TransformStream and SSE.
**When to use:** After tool_use loop completes, stream the final text response for chat UX.
**Example:**
```typescript
// Source: Next.js API route streaming pattern
import { NextRequest, NextResponse } from 'next/server'
import Anthropic from '@anthropic-ai/sdk'

export async function POST(request: NextRequest) {
  const { query, messages: chatMessages } = await request.json()

  // Build context, validate user, etc.
  const systemPrompt = await buildSystemPrompt(orgId, userRole)

  const encoder = new TextEncoder()
  const stream = new TransformStream()
  const writer = stream.writable.getWriter()

  // Start streaming in background
  ;(async () => {
    try {
      const messageStream = client.messages.stream({
        model: 'claude-sonnet-4-20250514',
        max_tokens: 1024,
        system: systemPrompt,
        tools,
        messages: [{ role: 'user', content: query }],
      })

      messageStream.on('text', (text) => {
        writer.write(encoder.encode(`data: ${JSON.stringify({ type: 'text', text })}\n\n`))
      })

      const finalMessage = await messageStream.finalMessage()

      // If tool_use, handle loop then stream final response
      // Send done event
      writer.write(encoder.encode(`data: ${JSON.stringify({ type: 'done', usage: finalMessage.usage })}\n\n`))
    } catch (error) {
      writer.write(encoder.encode(`data: ${JSON.stringify({ type: 'error', message: 'Marie encountered an error' })}\n\n`))
    } finally {
      writer.close()
    }
  })()

  return new NextResponse(stream.readable, {
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache, no-transform',
      Connection: 'keep-alive',
    },
  })
}
```

### Pattern 4: Smart Routing Scoring Algorithm
**What:** Score and rank drivers for a given load based on 5 weighted factors.
**When to use:** POST /api/dispatch/suggest endpoint.
**Example:**
```typescript
interface ScoringFactors {
  proximity: number      // 0-1 score, weight 0.30
  availability: number   // 0-1 score, weight 0.25
  equipment: number      // 0-1 score, weight 0.20
  performance: number    // 0-1 score, weight 0.15
  lane: number           // 0-1 score, weight 0.10
}

interface DriverSuggestion {
  driver_id: string
  driver_name: string
  score: number          // 0-100 weighted score
  factors: ScoringFactors
}

function calculateProximity(driverLastCity: string, driverLastState: string, pickupCity: string, pickupState: string): number {
  // Same state + same city = 1.0
  // Same state + different city = 0.7
  // Adjacent state = 0.4
  // Far state = 0.1
  if (driverLastCity?.toLowerCase() === pickupCity?.toLowerCase() &&
      driverLastState?.toLowerCase() === pickupState?.toLowerCase()) return 1.0
  if (driverLastState?.toLowerCase() === pickupState?.toLowerCase()) return 0.7
  // For adjacent states, maintain a simple adjacency map
  return 0.2 // Default for unknown proximity
}

function calculateScore(factors: ScoringFactors): number {
  return Math.round(
    (factors.proximity * 30 +
     factors.availability * 25 +
     factors.equipment * 20 +
     factors.performance * 15 +
     factors.lane * 10)
  )
}
```

### Anti-Patterns to Avoid
- **Sending entire database to Claude:** Limit context to recent/relevant data (last 50 loads, all active drivers). Token costs and latency grow linearly with context size.
- **Streaming tool_use blocks to the client:** Tool use requires a server-side loop. Only stream the final text response after all tools complete. Do not expose tool internals to the UI.
- **Storing conversation history:** Marie is stateless by design. Do not build a messages table or try to maintain conversation threads across requests.
- **Calling server actions directly from the API route:** Server actions use `formData` and `redirect()`. Create separate utility functions that Marie's tools can call without the Next.js server action wrappers.
- **Using GPS coordinates for proximity:** The PRD specifies city/state approximation. Do not add geocoding dependencies.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Claude API integration | HTTP fetch wrapper | @anthropic-ai/sdk | Handles streaming, retries, error types, TypeScript types |
| Tool input validation | Manual JSON parsing | Zod schemas + `strict: true` on tool definitions | Anthropic supports structured outputs for guaranteed schema match |
| SSE parsing on client | Manual EventSource parsing | Native EventSource or fetch with ReadableStream reader | Browser-native, handles reconnection |
| Rate limiting | Custom token bucket | Simple in-memory counter with sliding window | Phase 5 MVP doesn't need distributed rate limiting |
| State adjacency map | Geocoding API | Static adjacency lookup object | ~50 states, small constant map, no external API needed |

**Key insight:** The heaviest lifting is done by the Claude API itself. Marie's server code is primarily plumbing: fetch org data, build system prompt, define tools, proxy to Claude, stream response back. Keep it thin.

## Common Pitfalls

### Pitfall 1: Tool Use Loop Never Terminating
**What goes wrong:** Claude keeps calling tools in a loop without reaching end_turn.
**Why it happens:** Ambiguous tool descriptions or system prompt instructions that encourage repeated tool calls.
**How to avoid:** Set a maximum loop iteration count (3-5). If exceeded, return the accumulated response and note the limitation. Use clear tool descriptions that specify exactly when each tool should be used.
**Warning signs:** Latency > 15 seconds on a Marie query, token usage > 4000 on a single query.

### Pitfall 2: RLS Context Loss in API Route
**What goes wrong:** Marie's database queries bypass RLS because the API route uses the wrong Supabase client.
**Why it happens:** Using `supabaseAdmin` (service role key) instead of the server client with user auth cookies.
**How to avoid:** Always use `createClient()` from `@/lib/supabase/server` in the Marie API route. This carries the user's auth cookies and enforces RLS. The admin client should NEVER be used for Marie queries.
**Warning signs:** Marie returning data from other organizations.

### Pitfall 3: Streaming Response Not Starting
**What goes wrong:** Client sees no response until the entire tool_use loop completes.
**Why it happens:** The API route awaits the full tool loop before returning any response.
**How to avoid:** Two-phase approach: (1) Run the tool loop synchronously to completion, (2) Stream only the final text response. OR send an initial SSE event like `{ type: 'thinking' }` immediately so the UI can show a loading state.
**Warning signs:** 10+ second delay before any client response.

### Pitfall 4: Server Action FormData Incompatibility
**What goes wrong:** Marie's tools try to call `createLoad(formData)` but can't construct proper FormData in an API route.
**Why it happens:** Existing server actions expect `FormData` from form submissions, not JSON.
**How to avoid:** Extract the core logic from server actions into shared utility functions that accept plain objects. Marie tools call the utility functions, not the server actions directly.
**Warning signs:** TypeScript errors about FormData not being available; "redirect() called in API route" crashes.

### Pitfall 5: System Prompt Token Explosion
**What goes wrong:** System prompt with all org data exceeds token limits, costs spike.
**Why it happens:** Dumping entire tables into the system prompt without limits.
**How to avoid:** Limit data to: 20 most recent loads, all active drivers (usually < 50), 20 recent invoices. Use compact JSON format (no pretty-printing). Monitor token usage in marie_queries table.
**Warning signs:** tokens_used > 8000 consistently, latency_ms > 5000.

### Pitfall 6: Marie Button Not Visible on All Pages
**What goes wrong:** Marie button only appears on some pages.
**Why it happens:** Added to individual page components instead of the layout.
**How to avoid:** Add the Marie floating button component to `src/app/(app)/layout.tsx` and `src/app/driver/layout.tsx`. It renders at layout level, visible on every page within that layout.
**Warning signs:** User navigates to a page and Marie button disappears.

### Marie API Route (Complete Pattern)
```typescript
// src/app/api/marie/query/route.ts
import { NextRequest, NextResponse } from 'next/server'
import Anthropic from '@anthropic-ai/sdk'
import { createClient } from '@/lib/supabase/server'
import { buildSystemPrompt } from '@/lib/marie/system-prompt'
import { marieTools, executeTool } from '@/lib/marie/tools'

const anthropic = new Anthropic() // Uses ANTHROPIC_API_KEY env var

export async function POST(request: NextRequest) {
  const startTime = Date.now()
  const supabase = await createClient()

  // Validate user
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })

  const { data: profile } = await supabase
    .from('profiles')
    .select('org_id, role')
    .eq('id', user.id)
    .single()
  if (!profile?.org_id) return NextResponse.json({ error: 'No organization' }, { status: 403 })

  const { query } = await request.json()

  // Build context
  const systemPrompt = await buildSystemPrompt(supabase, profile.role)

  // Filter tools by role
  const availableTools = profile.role === 'driver' ? [] : marieTools

  // Tool use loop
  let messages: Anthropic.MessageParam[] = [{ role: 'user', content: query }]
  let response: Anthropic.Message
  let loopCount = 0
  const MAX_LOOPS = 5

  do {
    response = await anthropic.messages.create({
      model: 'claude-sonnet-4-20250514',
      max_tokens: 1024,
      system: systemPrompt,
      tools: availableTools,
      messages,
    })

    if (response.stop_reason === 'tool_use') {
      messages.push({ role: 'assistant', content: response.content })
      const toolResults: Anthropic.ToolResultBlockParam[] = []

      for (const block of response.content) {
        if (block.type === 'tool_use') {
          const result = await executeTool(block.name, block.input, supabase, profile.org_id)
          toolResults.push({
            type: 'tool_result',
            tool_use_id: block.id,
            content: JSON.stringify(result),
          })
        }
      }
      messages.push({ role: 'user', content: toolResults })
    }
    loopCount++
  } while (response.stop_reason === 'tool_use' && loopCount < MAX_LOOPS)

  // Extract text response
  const responseText = response.content
    .filter(b => b.type === 'text')
    .map(b => b.text)
    .join('')

  // Log query
  const latencyMs = Date.now() - startTime
  await supabase.from('marie_queries').insert({
    org_id: profile.org_id,
    user_id: user.id,
    query_text: query,
    response_text: responseText,
    query_type: response.stop_reason === 'end_turn' ? 'question' : 'action',
    tokens_used: (response.usage?.input_tokens ?? 0) + (response.usage?.output_tokens ?? 0),
    latency_ms: latencyMs,
    model: 'claude-sonnet-4-20250514',
    success: true,
  })

  return NextResponse.json({
    response: responseText,
    usage: response.usage,
  })
}
```

### Smart Routing Endpoint
```typescript
// src/app/api/dispatch/suggest/route.ts
import { NextRequest, NextResponse } from 'next/server'
import { createClient } from '@/lib/supabase/server'
import { scoreDriversForLoad } from '@/lib/routing/scoring'

export async function POST(request: NextRequest) {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })

  const { load_id } = await request.json()

  // Fetch load details
  const { data: load } = await supabase
    .from('loads')
    .select('*')
    .eq('id', load_id)
    .single()

  if (!load) return NextResponse.json({ error: 'Load not found' }, { status: 404 })

  const suggestions = await scoreDriversForLoad(supabase, load)

  return NextResponse.json({ suggestions })
}
```

### Chat Panel Client Component (SSE Consumer)
```typescript
// Pattern for consuming Marie responses on the client
'use client'
import { useState, useCallback } from 'react'

interface MarieMessage {
  role: 'user' | 'assistant'
  content: string
}

function useMarie() {
  const [messages, setMessages] = useState<MarieMessage[]>([])
  const [isLoading, setIsLoading] = useState(false)

  const sendQuery = useCallback(async (query: string) => {
    setMessages(prev => [...prev, { role: 'user', content: query }])
    setIsLoading(true)

    try {
      const res = await fetch('/api/marie/query', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ query }),
      })

      const data = await res.json()
      setMessages(prev => [...prev, { role: 'assistant', content: data.response }])
    } catch (error) {
      setMessages(prev => [...prev, {
        role: 'assistant',
        content: 'Sorry, I ran into an issue. Try again in a moment.'
      }])
    } finally {
      setIsLoading(false)
    }
  }, [])

  return { messages, isLoading, sendQuery }
}
```

### Inline Action Button Parsing
```typescript
// Parse action markers from Marie's response text
// Format: [ACTION:type:entity_id:label]
const ACTION_REGEX = /\[ACTION:(\w+):([a-f0-9-]+):([^\]]+)\]/g

interface ActionButton {
  type: 'view_load' | 'dispatch_driver' | 'generate_invoice'
  entityId: string
  label: string
}

function parseActions(text: string): { cleanText: string; actions: ActionButton[] } {
  const actions: ActionButton[] = []
  const cleanText = text.replace(ACTION_REGEX, (_, type, entityId, label) => {
    actions.push({ type, entityId, label })
    return '' // Remove from display text
  })
  return { cleanText: cleanText.trim(), actions }
}
```

| claude-3-sonnet model ID | claude-sonnet-4-20250514 | 2025 | Newer model with better tool use accuracy |

**Model recommendation:** Use `claude-sonnet-4-20250514` for Marie. It provides excellent tool use accuracy at reasonable cost. Haiku would be cheaper but less reliable for complex tool calls. The model string should be configurable via environment variable for easy upgrades.

## Open Questions

1. **Streaming vs JSON response for MVP**
   - What we know: Streaming provides better UX for longer responses
   - What's unclear: Whether the complexity of SSE + tool_use loop is worth it for MVP
   - Recommendation: Start with JSON response (simpler). Add streaming in a follow-up if response latency is noticeable. Tool_use loop requires non-streaming anyway, so streaming only helps the final text response.

2. **Rate limiting granularity**
   - What we know: Need to prevent API cost runaway
   - What's unclear: Per-user vs per-org limits, exact thresholds
   - Recommendation: Simple per-user rate limit of 20 queries per minute using in-memory Map with TTL. Log to marie_queries table for monitoring. Can tighten later based on usage data.

3. **Driver last known location for proximity**
   - What we know: Need driver's last delivery city/state for proximity scoring
   - What's unclear: Best query to determine this
   - Recommendation: Query the most recent completed dispatch for each driver, join to load's delivery_city/delivery_state. If no completed dispatch, use driver.home_terminal as fallback.

4. **MARI-09 alert summary scope**
   - What we know: proactive_alerts table needs to exist for Phase 6 wiring
   - What's unclear: Whether to create the table now or in Phase 6
   - Recommendation: Create the proactive_alerts table and RLS in Phase 5 migration (needed for MARI-09 placeholder). Leave it empty -- Phase 6 populates it via edge functions.
