# Phase 7: Compliance - Research

**Researched:** 2026-03-25
**Domain:** DOT/non-DOT regulatory compliance tracking, DVIR, DQ files, IFTA, compliance health scoring
**Confidence:** HIGH

## Summary

Phase 7 adds a comprehensive compliance tracking module to Manifest. The domain spans six database tables (compliance_profiles, compliance_items, compliance_alerts, driver_qualifications, inspections, ifta_records), a daily pg_cron compliance scanner function, and UI pages across all three modes (Command, Driver PWA, Owner-Operator). The compliance module adapts based on carrier type -- DOT-regulated carriers see the full suite (DQ files, annual inspections, IFTA) while non-DOT carriers see a subset (insurance, registration, vehicle inspections).

The PRD-03 Section 2 provides complete SQL schemas for all tables, and the existing pg_cron alert generator pattern from Phase 6 (migration 00019) is the proven template for the compliance-scanner function. Compliance alerts integrate with the existing proactive_alerts system, enabling Marie visibility and push notifications through infrastructure already built in Phases 5-6.

**Primary recommendation:** Build database/types first, then compliance scanner function, then UI pages layered on top. DVIR forms should optimize for speed (drivers fill these daily). IFTA is manual entry only for v1 (GPS/ELD deferred).

### Pattern 1: Compliance Scanner (pg_cron SQL Function)
**What:** Daily SQL function that scans compliance_items, generates alerts, updates statuses, and creates next recurrence
**When to use:** Daily at 6 AM via pg_cron
**Example:**
```sql
-- Follows exact pattern from 00019_alert_generators.sql
create or replace function check_compliance_items()
returns void
language plpgsql
security definer
set search_path = ''
as $$
declare
  _rec record;
  _days_until integer;
  _alert_type text;
  _severity text;
begin
  for _rec in
    select ci.*, cp.org_id
    from public.compliance_items ci
    join public.compliance_profiles cp on cp.id = ci.compliance_profile_id
    where ci.status not in ('completed', 'waived', 'not_applicable')
      and ci.due_date is not null
  loop
    _days_until := _rec.due_date - current_date;

    -- Update status based on days remaining
    if _days_until < 0 then
      update public.compliance_items set status = 'overdue', updated_at = now()
        where id = _rec.id and status != 'overdue';
    elsif _days_until <= 30 then
      update public.compliance_items set status = 'due_soon', updated_at = now()
        where id = _rec.id and status != 'due_soon';
    end if;

    -- Generate alerts at threshold days (de-duplicate with NOT EXISTS)
    if _days_until = any(_rec.alert_days_before) or _days_until < 0 then
      -- Determine alert_type and severity
      if _days_until < 0 then
        _alert_type := 'overdue'; _severity := 'critical';
      elsif _days_until <= 7 then
        _alert_type := 'due_soon'; _severity := 'warning';
      else
        _alert_type := 'approaching'; _severity := 'info';
      end if;

      -- De-duplicate: daily window
      if not exists (
        select 1 from public.compliance_alerts ca
        where ca.compliance_item_id = _rec.id
          and ca.alert_type = _alert_type
          and ca.created_at > now() - interval '24 hours'
      ) then
        insert into public.compliance_alerts (
          org_id, compliance_item_id, alert_type, days_until_due, message
        ) values (
          _rec.org_id, _rec.id, _alert_type, _days_until,
          _rec.title || ' is ' || case
            when _days_until < 0 then abs(_days_until) || ' days overdue'
            when _days_until = 0 then 'due today'
            else 'due in ' || _days_until || ' days'
          end
        );

        -- Also write to proactive_alerts for Marie visibility
        insert into public.proactive_alerts (
          org_id, alert_type, severity, title, message,
          related_entity_type, related_entity_id
        ) values (
          _rec.org_id, 'compliance_' || _alert_type, _severity,
          'Compliance: ' || _rec.title,
          _rec.title || ' is ' || case
            when _days_until < 0 then abs(_days_until) || ' days overdue'
            when _days_until = 0 then 'due today'
            else 'due in ' || _days_until || ' days'
          end,
          'compliance_item', _rec.id
        );
      end if;
    end if;
  end loop;

  -- Auto-generate next occurrence for completed recurring items
  for _rec in
    select ci.*
    from public.compliance_items ci
    where ci.status = 'completed'
      and ci.completed_date is not null
      and ci.recurrence_rule is not null
      and ci.recurrence_rule != 'custom'
      -- Only if no future item of same category exists
      and not exists (
        select 1 from public.compliance_items ci2
        where ci2.compliance_profile_id = ci.compliance_profile_id
          and ci2.category = ci.category
          and ci2.vehicle_id is not distinct from ci.vehicle_id
          and ci2.driver_id is not distinct from ci.driver_id
          and ci2.due_date > ci.due_date
          and ci2.id != ci.id
      )
  loop
    insert into public.compliance_items (
      org_id, compliance_profile_id, category, title, description,
      due_date, status, assigned_to, vehicle_id, driver_id,
      recurrence_rule, recurrence_months, alert_days_before
    ) values (
      _rec.org_id, _rec.compliance_profile_id, _rec.category,
      _rec.title, _rec.description,
      _rec.completed_date + case _rec.recurrence_rule
        when 'annual' then interval '1 year'
        when 'biennial' then interval '2 years'
        when 'quarterly' then interval '3 months'
        when 'monthly' then interval '1 month'
        when 'custom' then (_rec.recurrence_months || ' months')::interval
        else interval '1 year'
      end,
      'upcoming', _rec.assigned_to, _rec.vehicle_id, _rec.driver_id,
      _rec.recurrence_rule, _rec.recurrence_months, _rec.alert_days_before
    );
  end loop;
end;
$$;

select cron.schedule(
  'check-compliance-items',
  '0 6 * * *',
  $$SELECT check_compliance_items();$$
);
```

### Pattern 2: Recurrence Auto-Generation
**What:** When a compliance item is marked complete and has a recurrence_rule, automatically create the next occurrence
**When to use:** Both in the scanner (for batch processing) and in the server action (for immediate feedback)
**Logic:**
- `annual` -> due_date = completed_date + 1 year
- `biennial` -> due_date = completed_date + 2 years
- `quarterly` -> due_date = completed_date + 3 months
- `monthly` -> due_date = completed_date + 1 month
- `custom` -> due_date = completed_date + recurrence_months months

### Pattern 3: Health Score Calculation
**What:** 0-100 composite score reflecting org compliance posture
**Algorithm:**
```typescript
function calculateHealthScore(metrics: {
  totalItems: number
  overdueItems: number
  dueSoonItems: number      // within 30 days
  totalDrivers: number
  driversWithCompleteDQ: number
  totalVehicles: number
  vehiclesWithCurrentInspection: number  // annual DOT within 12 months
}): number {
  // Weight: overdue items 40%, due-soon items 15%, DQ completeness 25%, inspection currency 20%

  // Overdue penalty (40 points max)
  const overdueScore = metrics.totalItems > 0
    ? Math.max(0, 40 - (metrics.overdueItems / metrics.totalItems) * 40 * 3)
    : 40  // No items = full score

  // Due soon penalty (15 points max)
  const dueSoonScore = metrics.totalItems > 0
    ? Math.max(0, 15 - (metrics.dueSoonItems / metrics.totalItems) * 15 * 2)
    : 15

  // DQ completeness (25 points max)
  const dqScore = metrics.totalDrivers > 0
    ? (metrics.driversWithCompleteDQ / metrics.totalDrivers) * 25
    : 25

  // Inspection currency (20 points max)
  const inspectionScore = metrics.totalVehicles > 0
    ? (metrics.vehiclesWithCurrentInspection / metrics.totalVehicles) * 20
    : 20

  return Math.round(overdueScore + dueSoonScore + dqScore + inspectionScore)
}
```

### Pattern 4: DQ File Completeness
**What:** Calculate percentage of required documents present for each driver
**Required documents for DOT carriers (10 items per FMCSA 49 CFR Part 391):**
1. Employment application (application_date not null)
2. MVR (mvr_last_pulled not null and within 365 days)
3. Road test certificate (road_test_date not null)
4. CDL copy (cdl_number not null)
5. Medical certificate (medical_card_expiry not null and not expired)
6. Annual review of driving record (annual_review_date not null and within 365 days)
7. Drug test result (drug_test_last_date not null)
8. Previous employer verification (hire_date not null -- tracked as part of onboarding)
9. Driver acknowledgment of regulations (application_date not null)
10. Endorsements documented (endorsements array not empty OR no endorsements required)

```typescript
function calculateDQCompleteness(dq: DriverQualification): {
  percentage: number
  missing: string[]
} {
  const required = [
    { field: 'application_date', label: 'Employment Application' },
    { field: 'cdl_number', label: 'CDL Copy' },
    { field: 'medical_card_expiry', label: 'Medical Certificate', dateCheck: true },
    { field: 'mvr_last_pulled', label: 'Motor Vehicle Record', annualCheck: true },
    { field: 'road_test_date', label: 'Road Test Certificate' },
    { field: 'annual_review_date', label: 'Annual Review', annualCheck: true },
    { field: 'drug_test_last_date', label: 'Drug Test Result' },
    { field: 'hire_date', label: 'Employment Verification' },
  ]

  const missing: string[] = []
  let present = 0

  for (const req of required) {
    const value = dq[req.field as keyof DriverQualification]
    if (!value) {
      missing.push(req.label)
    } else if (req.dateCheck && new Date(value as string) < new Date()) {
      missing.push(`${req.label} (expired)`)
    } else if (req.annualCheck) {
      const lastDate = new Date(value as string)
      const oneYearAgo = new Date()
      oneYearAgo.setFullYear(oneYearAgo.getFullYear() - 1)
      if (lastDate < oneYearAgo) {
        missing.push(`${req.label} (stale > 1 year)`)
      } else {
        present++
      }
    } else {
      present++
    }
  }

  return {
    percentage: Math.round((present / required.length) * 100),
    missing,
  }
}
```

### Pattern 5: DOT vs Non-DOT Adaptation
**What:** Show/hide compliance modules based on carrier type
**Logic:**
- DOT carriers (`is_dot_regulated = true`): Full suite -- DQ files, annual DOT inspection, DVIR, IFTA, drug testing, operating authority
- Non-DOT carriers (`is_dot_regulated = false`): Subset -- insurance, vehicle registration, state inspections, basic vehicle safety checks
- Both: All modules visible
- Implementation: Server-side filtering in compliance actions + conditional UI rendering

### Anti-Patterns to Avoid
- **Hardcoding alert thresholds:** Use the `alert_days_before` integer array on each compliance_item. Different items need different warning periods (insurance = 90 days, medical card = 60 days, DVIR = N/A).
- **Client-side health score calculation:** Calculate on the server. The score aggregates across multiple tables and should not require multiple client-side fetches.
- **Blocking DVIR form with too many required fields:** FMCSA requires only 11 items. Keep the form fast -- drivers do this twice daily. Use pass/fail toggles, not text fields.
- **Mixing compliance_alerts and proactive_alerts:** compliance_alerts is the compliance-specific detail table. proactive_alerts is the cross-module surface for Marie and dashboard. The scanner writes to BOTH.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Date recurrence | Custom interval math | PostgreSQL interval arithmetic | `completed_date + interval '1 year'` handles leap years, month boundaries |
| CSV export | Custom string concatenation | Built-in array-to-CSV or simple map+join | Edge cases in quoting, escaping |
| File upload | New upload component | Existing `FileUpload` component | Already handles camera capture for mobile |
| Status badges | New badge component | Extend existing `StatusBadge` with 'compliance' variant | Consistent styling across app |

| Push notifications | New push logic | Existing `sendPushToOrgAdminsAndDispatchers` | Fire-and-forget pattern already proven |

## Common Pitfalls

### Pitfall 1: Recurrence Duplication
**What goes wrong:** Completing an item twice (race condition or double-click) creates duplicate next-occurrence items
**Why it happens:** No uniqueness constraint on next occurrence generation
**How to avoid:** Use NOT EXISTS check before inserting next occurrence (check for same category + vehicle/driver + due_date > current item's due_date). The scanner already does this.
**Warning signs:** Multiple future items of same category for same entity

### Pitfall 2: Scanner Generating Stale Alerts
**What goes wrong:** Scanner creates alerts for items already marked completed between runs
**Why it happens:** Completion happened after last scan but before next scan
**How to avoid:** Always filter `status not in ('completed', 'waived', 'not_applicable')` in scanner query

### Pitfall 3: DVIR Form Performance on Mobile
**What goes wrong:** Form takes too long to fill out, drivers skip it
**Why it happens:** Too many text fields, complex validation, heavy page load
**How to avoid:** Use toggle buttons (pass/fail) for each of the 11 FMCSA items. Only show text fields for items marked "fail" (for defect description). Pre-populate vehicle info from driver's assigned vehicle.
**Warning signs:** Form completion time > 2 minutes

### Pitfall 4: IFTA Tax Rate Staleness
**What goes wrong:** Tax calculations use outdated rates
**Why it happens:** IFTA tax rates change quarterly per jurisdiction
**How to avoid:** Store `tax_rate` per ifta_record row (point-in-time). Do NOT store a global tax rate table. The user enters the rate for their filing quarter.
**Warning signs:** Tax calculations don't match filed returns

### Pitfall 5: Compliance Profile Not Created
**What goes wrong:** User navigates to /compliance but has no compliance_profile
**Why it happens:** Profile is a separate entity from organization
**How to avoid:** Auto-create compliance_profile when user first visits /compliance, pre-populated from organizations table (DOT number, MC number, company_type). Or use an "onboarding" step.
**Warning signs:** 404 or empty state on first visit

### Pitfall 6: DQ Completeness Drift
**What goes wrong:** DQ file shows 100% but documents have expired since last check
**Why it happens:** Completeness is checked at query time but not re-validated against expiry dates
**How to avoid:** Always compute DQ completeness dynamically (not cached) and check date validity for MVR (annual) and medical card (expiry date)

### Compliance Item Server Action
```typescript
// Source: Project conventions from src/lib/alerts/actions.ts
'use server'

import { createClient } from '@/lib/supabase/server'
import { revalidatePath } from 'next/cache'

export async function completeComplianceItem(
  itemId: string,
  documentUrls?: string[]
): Promise<{ error?: string }> {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated' }

  const completedDate = new Date().toISOString().split('T')[0]

  // Update item
  const { data: item, error } = await supabase
    .from('compliance_items')
    .update({
      status: 'completed',
      completed_date: completedDate,
      document_urls: documentUrls ?? [],
      updated_at: new Date().toISOString(),
    })
    .eq('id', itemId)
    .select('*, compliance_profiles!inner(org_id)')
    .single()

  if (error || !item) return { error: error?.message ?? 'Item not found' }

  // Auto-generate next occurrence if recurring
  if (item.recurrence_rule) {
    const nextDueDate = calculateNextDueDate(completedDate, item.recurrence_rule, item.recurrence_months)
    await supabase.from('compliance_items').insert({
      org_id: item.org_id,
      compliance_profile_id: item.compliance_profile_id,
      category: item.category,
      title: item.title,
      description: item.description,
      due_date: nextDueDate,
      status: 'upcoming',
      assigned_to: item.assigned_to,
      vehicle_id: item.vehicle_id,
      driver_id: item.driver_id,
      recurrence_rule: item.recurrence_rule,
      recurrence_months: item.recurrence_months,
      alert_days_before: item.alert_days_before,
    })
  }

  revalidatePath('/compliance')
  return {}
}

function calculateNextDueDate(
  fromDate: string,
  rule: string,
  customMonths?: number
): string {
  const date = new Date(fromDate)
  switch (rule) {
    case 'annual': date.setFullYear(date.getFullYear() + 1); break
    case 'biennial': date.setFullYear(date.getFullYear() + 2); break
    case 'quarterly': date.setMonth(date.getMonth() + 3); break
    case 'monthly': date.setMonth(date.getMonth() + 1); break
    case 'custom': date.setMonth(date.getMonth() + (customMonths ?? 12)); break
  }
  return date.toISOString().split('T')[0]
}
```

### IFTA Calculation
```typescript
// Source: IFTA standard formula (freightwaves.com, FMCSA)
export function calculateIFTA(records: IFTARecord[], totalFleetMiles: number, totalFleetGallons: number) {
  const avgMPG = totalFleetGallons > 0 ? totalFleetMiles / totalFleetGallons : 0

  return records.map(record => {
    const gallonsConsumed = avgMPG > 0 ? record.miles_traveled / avgMPG : 0
    const netTaxableGallons = gallonsConsumed - record.gallons_purchased
    const taxOwed = netTaxableGallons * (record.tax_rate ?? 0)

    return {
      ...record,
      gallons_consumed: Math.round(gallonsConsumed * 1000) / 1000,
      net_taxable_gallons: Math.round(netTaxableGallons * 1000) / 1000,
      tax_owed: Math.round(taxOwed * 100) / 100,
      net_tax: Math.round((taxOwed - (record.tax_paid ?? 0)) * 100) / 100,
    }
  })
}
```

## Open Questions

1. **Compliance profile auto-creation timing**
   - What we know: PRD says compliance_profiles links to org, but user must set carrier_type and DOT details
   - What's unclear: Should profile be created on first /compliance visit or require explicit setup?
   - Recommendation: Show a setup prompt on first visit that pre-fills from organizations table, then creates the profile. After that, always show dashboard.

2. **Alert type for proactive_alerts integration**
   - What we know: Existing AlertType is `'late_pickup' | 'driver_silent' | ...` (6 types)
   - What's unclear: How to add compliance alert types without breaking existing type
   - Recommendation: Extend AlertType union with compliance-specific types like `'compliance_overdue' | 'compliance_due_soon' | 'compliance_approaching'` or use a generic `'compliance'` type with details in the message.

3. **IFTA quarter format and deadlines**
   - What we know: IFTA uses '2026-Q1' format, filings due 30 days after quarter end
   - What's unclear: Whether to enforce filing deadline alerts as compliance_items
   - Recommendation: Auto-create quarterly IFTA compliance_items with recurrence_rule='quarterly' for carriers with IFTA license
