The analytics-builder pg_cron function computes daily/weekly/monthly snapshots.

2. CREATE TABLE driver_performance:
   - id UUID PRIMARY KEY DEFAULT gen_random_uuid()
   - org_id UUID NOT NULL REFERENCES organizations(id)
   - driver_id UUID NOT NULL REFERENCES drivers(id)
   - period_start DATE NOT NULL
   - period_end DATE NOT NULL
   - loads_completed INTEGER DEFAULT 0
   - miles_driven INTEGER DEFAULT 0
   - revenue_generated NUMERIC(12,2) DEFAULT 0
   - on_time_pct NUMERIC(5,2)
   - fuel_efficiency NUMERIC(5,2) (mpg)
   - safety_incidents INTEGER DEFAULT 0
   - compliance_score INTEGER (0-100)
   - customer_complaints INTEGER DEFAULT 0
   - created_at TIMESTAMPTZ DEFAULT now()
   - UNIQUE(org_id, driver_id, period_start)
   - RLS enabled, org_isolation policy using auth.org_id()

3. CREATE the analytics-builder pg_cron function:
   - Runs daily at 1 AM UTC
   - For each org: queries loads, fuel_transactions, maintenance_records, compliance_items, crm_companies
   - Computes daily snapshot with all new fields (expenses = fuel + maintenance costs, profit = revenue - expenses, etc.)
   - On first day of week: aggregates weekly snapshot from daily records
   - On first day of month: aggregates monthly snapshot
   - Computes driver_performance for each active driver in the org (loads, miles, revenue from dispatches/loads, on-time from load_status_history, fuel from fuel_transactions, compliance from compliance_items)
   - Uses UPSERT (ON CONFLICT) for idempotency

4. Update src/types/database.ts:
   - Create new `AnalyticsSnapshot` type extending DailySnapshot with all new fields plus `period: 'daily' | 'weekly' | 'monthly'`
   - Keep DailySnapshot as alias for backward compat
   - Add `DriverPerformance` type matching the table
   - Add analytics_snapshots expanded entries to Database type (update existing daily_snapshots entry)
   - Add driver_performance to Database type
