Create migration with cross-module integration triggers:

**1. DOT inspection -> compliance auto-complete (XMOD-02)**

Create function `on_inspection_completed()`:
- Trigger: AFTER INSERT ON inspections
- When NEW.inspection_type = 'annual_dot' AND NEW.result IN ('pass', 'conditional'):
  - Find compliance_items WHERE vehicle_id = NEW.vehicle_id AND category = 'annual_inspection' AND status IN ('upcoming', 'due_soon', 'overdue')
  - UPDATE those items SET status = 'completed', completed_date = NEW.inspection_date
  - INSERT new compliance_item for next annual inspection: due_date = NEW.inspection_date + interval '1 year', category = 'annual_inspection', title = 'Annual DOT Inspection', status = 'upcoming', vehicle_id = NEW.vehicle_id, recurrence_rule = 'annual'
  - Get compliance_profile_id from existing item or lookup by org_id

CREATE TRIGGER inspection_completed_compliance AFTER INSERT ON inspections FOR EACH ROW WHEN (NEW.inspection_type = 'annual_dot') EXECUTE FUNCTION on_inspection_completed();

**2. CDL expiry cross-module flagging (XMOD-03)**

Create function `check_cdl_expiry()`:
- Runs as part of compliance_scanner (extend existing 00022 scanner or add new cron)
- Alternative: Add to follow_up_reminder cron at 7 AM
- SELECT driver_qualifications WHERE cdl_expiry <= CURRENT_DATE + interval '30 days' AND cdl_expiry > CURRENT_DATE - interval '1 day'
- For each expiring/expired CDL:
  - INSERT proactive_alert (alert_type='compliance_due_soon', severity based on days remaining, title='CDL Expiring', message with driver name and expiry date, related_entity_type='driver', related_entity_id=driver_id)
  - De-duplicate with NOT EXISTS (same alert_type, related_entity_id, within 24 hours)
  - Check for active dispatches for this driver: SELECT dispatches WHERE driver_id = dq.driver_id AND status NOT IN ('completed', 'rejected')
  - If active dispatches exist: INSERT additional alert (severity='critical', title='CDL Expiring - Active Loads', message listing load numbers)

Schedule: `SELECT cron.schedule('cdl-expiry-checker', '0 7 * * *', 'SELECT check_cdl_expiry()')`

**3. Fuel -> IFTA feed verification (XMOD-04)**

This is already connected from Phase 7/8 (fuel_transactions feed IFTA). Add a verification comment and ensure the relationship is documented. No new trigger needed -- the existing IFTA entry flow already uses fuel transactions. Add a SQL comment documenting the data flow.

**4. Marie CRM/compliance check hooks (XMOD-05)**

Create two SQL functions that Marie's tools can call:
- `get_company_payment_history(p_org_id UUID, p_company_name TEXT)` RETURNS TABLE (company_name TEXT, days_to_pay INTEGER, credit_score TEXT, total_revenue NUMERIC, total_loads INTEGER, payment_terms TEXT)
  - Matches company by name (ilike) within org, returns payment info
- `get_driver_compliance_status(p_org_id UUID, p_driver_id UUID)` RETURNS TABLE (driver_name TEXT, cdl_expiry DATE, medical_card_expiry DATE, dq_file_complete BOOLEAN, overdue_items INTEGER, active_alerts INTEGER)
  - Joins driver_qualifications + compliance_items + proactive_alerts for driver status

These functions can be called from Marie's tool system via supabase.rpc().
