    **00018_daily_snapshots.sql:**
    - CREATE TABLE daily_snapshots per PRD-02 Section 5.2: id (uuid pk), org_id (fk), snapshot_date (date), loads_booked, loads_delivered, loads_canceled (int), revenue, total_miles, revenue_per_mile (numeric), on_time_deliveries, total_deliveries, on_time_percentage (numeric), active_drivers, invoices_generated (int), invoices_paid (numeric), created_at. UNIQUE(org_id, snapshot_date).
    - RLS: enable, SELECT policy for org members using (select org_id from profiles where id = auth.uid())
    - CREATE FUNCTION generate_daily_snapshot() per research code example -- loops orgs, aggregates previous day from loads/load_status_history/invoices/drivers, ON CONFLICT upsert. SECURITY DEFINER.
    - On-time: delivery is "on-time" if load_status_history entry for new_status='delivered' created_at::date <= delivery_date. Count on_time_deliveries and total_deliveries, compute percentage.
    - pg_cron: schedule 'daily-snapshot-generator' at '0 1 * * *'

    **00019_alert_generators.sql:**
    - 5 SQL functions (overdue invoice enhancement is just extending existing pg_cron output):
      1. check_late_pickup(): loads with status in (dispatched, in_transit) joined to dispatches, where pickup_date::timestamp - now() < interval '3 hours' AND pickup_date::timestamp > now(). Use driver's last delivery state vs pickup state adjacency as proximity heuristic (same state = ok, adjacent = warning, non-adjacent = critical). De-duplicate with NOT EXISTS check on proactive_alerts for same load + alert_type in last 30 min. Schedule: every 30 min.
      2. check_driver_silent(): loads with active dispatches where status in (in_transit, at_pickup, loaded, at_delivery) and no load_status_history entry in last 4 hours. Warning at 4h, critical at 8h. De-duplicate 1 hour window. Schedule: every hour.
      3. check_overdue_invoices_alerts(): invoices where status='sent' and due_date < CURRENT_DATE. Warning 1-7 days overdue, critical 7+. This SUPPLEMENTS the existing overdue scanner (which transitions status), by also writing proactive_alerts. De-duplicate daily. Schedule: daily 8:05 AM (5 min after existing scanner).
      4. check_eta_risk(): loads in_transit with dispatches, estimate delivery based on miles and average 50mph, compare to delivery_date. Warning if estimated > delivery window. De-duplicate 1 hour. Schedule: every hour.
      5. check_unassigned_loads(): booked loads with pickup < 24h and no active dispatch. Warning at 24h, critical at 12h. De-duplicate 2 hours. Schedule: every 2 hours.
    - Dispatch conflict (ALRT-04) is NOT a pg_cron function -- it runs on dispatch creation, handled in conflict-check.ts utility.
    - All functions: SECURITY DEFINER, proper de-duplication with NOT EXISTS.

    **00020_push_subscriptions.sql:**
    - CREATE TABLE push_subscriptions: id (uuid pk), user_id (fk auth.users ON DELETE CASCADE), org_id (fk organizations ON DELETE CASCADE), endpoint (text NOT NULL), keys_p256dh (text NOT NULL), keys_auth (text NOT NULL), created_at. UNIQUE(user_id, endpoint).
    - RLS: enable, policy for users to manage own subscriptions (user_id = auth.uid())
    - ALTER TABLE profiles ADD COLUMN notification_preferences jsonb DEFAULT '{"new_dispatch": true, "load_status_change": true, "critical_alert": true, "invoice_paid": true, "driver_response": true}'::jsonb;
